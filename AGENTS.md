# Repository Guidelines

## Project Structure & Module Organization
- `config/` holds Hugo configuration (TOML).
- `content/` contains published site content in Markdown (Hugo pages and sections).
- `content-org/` contains Org-mode source files used with `ox-hugo` (preferred for new/updated long-form posts).
- `layouts/` and `themes/` contain Hugo templates and theme overrides.
- `assets/` holds pipeline assets processed by Hugo Pipes; `static/` is copied as-is to the site.
- `public/` is the generated output (do not edit by hand).
- `data/`, `i18n/`, `archetypes/` contain Hugo data files, translations, and content templates.

## Build, Test, and Development Commands
- `mise run serve` runs `hugo serve --navigateToChanged` for local development.
- `mise run servedebug` runs `hugo serve -D --navigateToChanged` to include drafts.
- `hugo --gc --minify -b $URL` is the Netlify production build command (see `netlify.toml`).

## Coding Style & Naming Conventions
- Follow Hugo conventions for front matter and section layout.
- Keep Markdown and Org files consistent with existing files in the same section.
- Use descriptive, URL-friendly filenames for content (e.g., `content/posts/2024-01-15-title.md`).
- When editing templates or assets, mirror the existing formatting in the file.

## Testing Guidelines
- No automated tests are defined in this repository.
- Validate changes by running `mise run serve` and checking the rendered pages locally.
- For production parity, verify the site builds with the Netlify command above.

## Commit & Pull Request Guidelines
- Commit messages in this repo are short, sentence-style summaries (e.g., `Updated blowfish`, `New CV`).
- For pull requests, include:
  - A concise summary of the change.
  - Links to affected pages or sections.
  - Screenshots for layout or visual updates.

## Configuration & Deployment Notes
- Hugo version and build settings are pinned in `netlify.toml`.
- Do not edit `public/` directly; it is generated by Hugo.
